Q9: 吞吐量与QPS深度剖析 ​

很多面试者对吞吐量、QPS、TPS、CPS这些概念混为一谈，导致面试被问住。本文帮你彻底理清这些概念，并给出实用的参考数值。

核心概念辨析 ​

什么是QPS？ ​

QPS (Queries Per Second) = 每秒查询数

简单理解：服务器每秒能处理多少个请求

┌─────────────────────────────────────────────────────┐
│                    QPS                              │
│  ← 1秒 →                                           │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐        │
│  │ req │ │ req │ │ req │ │ req │ │ req │  ...   │
│  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘        │
│                                                      │
│  如果1秒处理了1000个请求 → QPS = 1000              │
└─────────────────────────────────────────────────────┘

什么是吞吐量？ ​

吞吐量 (Throughput) = 单位时间内处理的数据量

和QPS的关系：
- QPS = 每秒请求数（关注请求数量）
- 吞吐量 = 每秒处理数据量（关注数据量）

举例：
- QPS=1000，每个请求1KB → 吞吐量 = 1MB/s
- QPS=100，每个请求10MB → 吞吐量 = 1MB/s

常见混淆概念 ​

概念全称含义面试重点
QPSQueries Per Second每秒查询数单机处理能力
TPSTransactions Per Second每秒事务数业务完整性
RPSRequests Per Second每秒请求数同QPS
CPSConnections Per Second每秒新建连接数网络层面
RTResponse Time响应时间用户体验
PVPage View页面访问量统计口径
UVUnique Visitor独立访客统计口径

QPS vs TPS 关键区别 ​

QPS = 每秒HTTP请求数（针对接口调用）
TPS = 每秒数据库事务数（针对数据库操作）

重要澄清：
┌─────────────────────────────────────────────────────────────┐
│  QPS是针对"请求"的，一个HTTP请求 = 1 QPS                  │
│  TPS是针对"事务"的，一个数据库事务 = 1 TPS                │
│                                                             │
│  方法内部的多次查询不算QPS！                                │
└─────────────────────────────────────────────────────────────┘

场景解析：你的理解是正确的！ ​

以你描述的Spring Boot下单接口为例：

┌─────────────────────────────────────────────────────────────┐
│  HTTP请求: POST /order/create                              │
│    ↓                                                      │
│  QPS = 1 (一个请求)                                      │
│    ↓                                                      │
│  开启事务                                                 │
│    ↓                                                      │
│  查询库存 → 1次SQL                                        │
│  校验用户  → 1次SQL                                        │
│  创建订单  → 1次SQL                                        │
│    ↓                                                      │
│  提交事务                                                 │
│    ↓                                                      │
│  TPS = 1 (一个事务)                                      │
└─────────────────────────────────────────────────────────────┘

结论：
- 你接收一个下单请求 → QPS = 1
- 你开启一个事务 → TPS = 1
- 事务里的3次SQL查询 → 只是3次数据库操作，不算QPS！

之前的例子是我表述不清，特此更正。

什么情况下QPS会大于1？ ​

场景1：前端多次请求
┌─────────────────────────────────────────────────┐
│  前端: 页面加载 → 发起3个HTTP请求              │
│    GET /api/userinfo   (查)                   │
│    GET /api/products    (查)                  │
│    POST /api/cart      (写)                   │
│                                                 │
│  结果: QPS = 3, TPS可能是1-3                   │
└─────────────────────────────────────────────────┘

详细解释：

关键点：TPS是数据库事务数量，不是请求数量！

QPS = 3 (3个HTTP请求)

这3个请求的事务情况：

  1. GET /api/userinfo → 无事务 (只读，autocommit) → TPS + 0

  2. GET /api/products → 无事务 (只读，autocommit)
    → TPS + 0

  3. POST /api/cart → 有事务 (插入操作) → TPS + 1

结果：QPS = 3, TPS = 1

另一种情况：3个请求都开启了事务

  1. GET /api/userinfo → 有事务 (FOR UPDATE锁读)
  2. GET /api/products → 有事务 (FOR UPDATE锁读)
  3. POST /api/cart → 有事务 (INSERT)

结果：QPS = 3, TPS = 3

为什么查询请求通常没有事务？ ┌─────────────────────────────────────────────┐ │ Spring Boot默认配置： │ │ • autocommit = true │ │ • 每个SQL单独提交 │ │ • 不占用数据库连接资源 │ │ │ │ 只有显式开启事务或写操作才占用事务： │ │ • @Transactional │ │ • INSERT/UPDATE/DELETE │ │ • SELECT ... FOR UPDATE │ └─────────────────────────────────────────────┘


场景2：前端轮询
┌─────────────────────────────────────────────────┐
│  前端: 每5秒请求一次接口获取最新数据           │
│    GET /api/notification (每5秒)              │
│                                                 │
│  结果: 1分钟内120次请求 → QPS = 2              │
└─────────────────────────────────────────────────┘

场景3：微服务调用
┌─────────────────────────────────────────────────┐
│  用户服务 → 订单服务 → 库存服务               │
│    1个请求 → 3个内部RPC调用                   │
│                                                 │
│  结果: 对用户来说 QPS=1                        │
│       订单服务承受的QPS也=1                    │
└─────────────────────────────────────────────────┘

面试加分话术 ​

正确理解：
"我理解QPS是针对HTTP请求/响应的，一个接口调用算1个QPS。TPS是针对数据库事务的，一个事务算1个TPS。

比如我写的一个Spring Boot下单接口，用户调用一次POST /order/create，这是一个QPS。接口内部开启一个数据库事务执行查询、校验、下单，这是1个TPS。

所以QPS和TPS的关系取决于：
- 如果是前后端分离，一个页面发多个请求 → QPS > TPS
- 如果是后端聚合调用 → QPS < TPS
- 如果是简单CRUD单表操作 → QPS ≈ TPS
"

注意：以上是单机预估，实际会受到以下因素影响

影响因素有哪些？ ​

QPS瓶颈分析：

1. CPU计算型
   - 场景：JSON序列化/反序列化、加密解密
   - 瓶颈：CPU核心数
   - 优化：多线程、异步处理

2. IO密集型
   - 场景：数据库查询、文件读写、网络请求
   - 瓶颈：磁盘IO、网络延迟
   - 优化：缓存、连接池、异步IO

3. 内存型
   - 场景：大数据处理、复杂计算
   - 瓶颈：内存大小、GC频率
   - 优化：数据结构优化、减少对象创建

4. 网络型
   - 场景：大文件上传下载、高并发推送
   - 瓶颈：带宽、网卡
   - 优化：压缩、CDN、分片

多台机器：QPS如何扩展？ ​

理论计算 ​

单机 QPS = 1000
3台机器负载均衡 → 总QPS = 3000 (理想情况)

但实际上有损耗：
┌─────────────────────────────────────────────────────┐
│  实际QPS = 单机QPS × 机器数 × 负载均衡效率          │
│                                                     │
│  负载均衡效率通常为 70-90%                          │
│                                                     │
│  例子：                                             │
│  - 单机1000 QPS                                    │
│  - 3台机器                                         │
│  - 效率85%                                         │
│  - 实际 = 1000 × 3 × 0.85 = 2550 QPS             │
└─────────────────────────────────────────────────────┘

扩展方式：水平 vs 垂直 ​

垂直扩展 (Scale Up)：
┌─────────────────────────────────────────────────────┐
│  2核 → 4核 → 8核 → 16核                           │
│                                                     │
│  优点：简单、避免分布式复杂性                        │
│  缺点：有上限、成本非线性增长                       │
│  适用：小到中等规模                                 │
└─────────────────────────────────────────────────────┘

水平扩展 (Scale Out)：
┌─────────────────────────────────────────────────────┐
│  1台 → 3台 → 10台 → 100台                         │
│                                                     │
│  优点：理论上无限扩展                               │
│  缺点：分布式复杂性（一致性、事务、网络）            │
│  适用：大规模应用                                   │
└─────────────────────────────────────────────────────┘

实际案例：3台4核8G机器 ​

配置：3台 4核8G
业务：常规CRUD接口（有Redis缓存+MySQL）

理论计算：
- 单机预估：1500 QPS
- 3台负载均衡：1500 × 3 = 4500 QPS
- 考虑损耗(85%)：4500 × 0.85 ≈ 3800 QPS

实际建议：
- 日常QPS：控制在 2000 以下（留余量）
- 峰值QPS：可达 3500
- 超过4000：考虑扩容或优化

实战场景：如何根据QPS做调整？ ​

场景一：日常CRUD接口 ​

┌─────────────────────────────────────────────────────┐
│  场景：用户管理系统（增删改查）                     │
│  单机配置：4核8G                                   │
│  技术栈：Spring Boot + MySQL + Redis               │
└─────────────────────────────────────────────────────┘

瓶颈分析：

- Redis连接数
- 业务线程池

调整建议：
1. dbcp2连接池：10-20 → 30
2. Redis连接池：8 → 50

场景二：高并发查询 ​

┌─────────────────────────────────────────────────────┐
│  场景：商品详情页（读多写少）                        │

│  技术栈：Spring Boot + Redis缓存 + MySQL           │
└─────────────────────────────────────────────────────┘

预估QPS：5000-8000（缓存命中率高时）

瓶颈分析：
- Redis QPS能力很强（可达10W+）
- 网络带宽
- 序列化/反序列化

调整建议：
1. 使用Redis集群
2. 减少Redis网络开销（Pipeline）
3. 考虑本地缓存（Caffeine）

场景三：复杂计算型 ​

┌─────────────────────────────────────────────────────┐
│  场景：报表统计（复杂SQL+计算）                     │
│  单机配置：8核16G                                   │
│  技术栈：Spring Boot + MySQL + ES                  │
└─────────────────────────────────────────────────────┘

预估QPS：100-500

瓶颈分析：
- 数据库CPU
- 复杂计算

调整建议：
1. 异步处理：接单后返回，异步计算
2. 预计算：定时任务预先算好
3. 降级：高峰期关闭实时统计

优化思路：
不是提高QPS，而是降低QPS压力（改为批量/异步）

面试高频追问 ​

追问1：日活100万需要多少QPS？ ​

计算过程：
┌─────────────────────────────────────────────────────┐
│  100万日活                                         │
│  - 假设平均每人访问10次/天                          │
│  - PV = 100万 × 10 = 1000万                        │
│  - 假设80%访问集中在4小时                          │
│  - 每小时PV = 1000万 ÷ 4 = 250万                   │
│  - 每秒PV = 250万 ÷ 3600 ≈ 700                     │
│                                                     │
│  考虑峰值（3-5倍）：700 × 4 = 2800                │
│  留30%余量：2800 ÷ 0.7 ≈ 4000 QPS                 │
└─────────────────────────────────────────────────────┘

结论：
- 日活100万 → 预估峰值4000 QPS
- 3台4核8G机器即可满足
- 如果更复杂，需要5-8台

追问3：QPS和响应时间RT什么关系？ ​

公式：
QPS = 并发数 / RT(秒)

举例：
- 并发100个请求
- 每个请求RT=100ms
- QPS = 100 / 0.1 = 1000

优化RT就能提升QPS：

│  │  业务计算      ████                    10ms   │   │
│  │  网络传输      █████████████           30ms   │   │
│  │  序列化        ████                    10ms   │   │
│  │  ─────────────────────────────────────       │   │
│  │  总计          100ms                        │   │
│  └─────────────────────────────────────────────┘   │
│                                                     │
│  优化方向：先优化耗时最长的（数据库）                │
└─────────────────────────────────────────────────────┘

追问5：如何设置合理的线程数？ ​

公式：
线程数 = CPU核心数 × 目标CPU利用率 × (1 + 等待时间/计算时间)

举例：
- 4核CPU
- 目标CPU利用率：80%
- 业务：等待时间10ms，计算时间5ms (IO密集型)

线程数 = 4 × 0.8 × (1 + 10/5) = 4 × 0.8 × 3 = 9.6 ≈ 10

实际建议：
┌─────────────────────────────────────────────────────┐
│  CPU密集型：CPU核心数 + 1~2                        │
│  例如：4核 → 5-6个线程                             │
│                                                     │
│  IO密集型：CPU核心数 × 2                           │
│  例如：4核 → 8个线程                               │
│                                                     │
│  混合型：CPU核心数 × (1 + 等待/计算)               │
└─────────────────────────────────────────────────────┘

追问6：QPS vs 并发数的关系 ​

并发数 = QPS × 平均响应时间

举例：
- QPS = 1000
- 平均RT = 100ms = 0.1s
- 并发数 = 1000 × 0.1 = 100

实际意义：
- 并发100不代表同时有100个请求在处理
- 可能是QPS高，RT短
- 1000 QPS + 100ms RT = 并发100
- 100 QPS + 1s RT = 也是并发100

实际开发中的QPS优化 ​

日常开发如何考虑QPS？ ​

1. 接口设计阶段
   ┌─────────────────────────────────────────────┐
   │  • 查为主的接口 → 可接受高QPS               │
   │  • 写为主的接口 → 关注一致性               │
   │  • 复杂计算 → 考虑异步/批量               │
   └─────────────────────────────────────────────┘

2. 技术选型阶段
   ┌─────────────────────────────────────────────┐
   │  • QPS < 1000 → 单机MySQL + Redis           │
   │  • QPS < 10000 → 读写分离 + Redis           │
   │  • QPS > 10000 → 分库分表 + ES              │
   └─────────────────────────────────────────────┘

3. 上线评估阶段
   ┌─────────────────────────────────────────────┐
   │  • 预估QPS                                  │
   │  • 准备对应配置                             │
   │  • 制定扩容预案                             │
   └─────────────────────────────────────────────┘

代码层面的QPS优化 ​

java
// 优化前：高QPS下性能差
public User getUser(Long id) {
    // 每次都查数据库
    return userMapper.selectById(id);
}

// 优化后：使用缓存
public User getUser(Long id) {
    // 1. 先查本地缓存
    User user = localCache.get(id);
    if (user != null) {
        return user;
    }
    
    // 2. 再查Redis
    user = redis.get("user:" + id);
    if (user != null) {
        localCache.put(id, user);
        return user;
    }
    
    // 3. 最后查数据库
    user = userMapper.selectById(id);
    if (user != null) {
        redis.set("user:" + id, user);
        localCache.put(id, user);
    }
    return user;
}

面试加分总结 ​

"我在工作中会这样评估系统QPS：

首先，明确业务类型——查多写少还是写多查少，这决定了优化方向。然后，根据服务器配置给出一个预估：4核8G常规业务大概能抗1500-2000QPS。最后，我会预留30%的余量，因为线上流量往往比预估高30%-50%。

当单机达到瓶颈时，我的处理顺序是：先优化代码和缓存，其次考虑垂直升级，最后才是水平扩展。盲目加机器是最省事但也是最贵的方案。"
