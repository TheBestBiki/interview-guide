# Q7: 分布式事务如何解决？Seata的AT模式？

## 分布式事务解决方案

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 2PC | 两阶段提交 | 强一致 | 同步阻塞、单点故障 | 强一致性要求 |
| TCC | Try-Confirm-Cancel | 性能好 | 侵入性强 | 高性能、业务可控 |
| 可靠消息 | MQ事务消息 | 最终一致 | 有延迟 | 异步场景 |
| Saga | 正向逆向补偿 | 长事务 | 可能数据不一致 | 长流程 |
| AT模式 | 自动生成反向SQL | 无侵入 | 性能损耗 | 大多数场景 |

## Seata AT模式原理

### 三种角色
- **TM（Transaction Manager）**：事务管理器，开启全局事务
- **TC（Transaction Coordinator）**：事务协调者，协调各分支事务
- **RM（Resource Manager）**：资源管理器，执行SQL、注册分支事务

### 执行流程
1. TM向TC申请开启全局事务，生成XID（全局事务ID）
2. 各RM执行SQL：记录前后镜像、生成undo_log、注册分支事务、提交本地事务
3. RM向TC报告分支事务状态
4. 如果全部成功：TC发送分支提交，RM异步删除undo_log
5. 如果失败：TC发送分支回滚，RM根据undo_log生成反向SQL执行

### AT模式优点
- **无侵入**：自动生成反向SQL，业务代码无需修改
- **性能好**：本地事务一阶段提交
- **易用**：配置简单，接入成本低
