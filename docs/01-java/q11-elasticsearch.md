# Q11: Elasticsearch深度剖析

## 一、核心概念与架构

### 1.1 ES vs MySQL 对比

| MySQL | ES | 说明 |
|--------|-----|------|
| Database | Index | 索引 |
| Table | Type（7.x已废弃） | 类型 |
| Row | Document | 文档 |
| Column | Field | 字段 |
| Schema | Mapping | 映射 |
| SQL | Query DSL | 查询语言 |

**追问1：为什么7.x废弃Type？**

一个索引只能有一种类型（_doc），简化了架构，避免跨类型查询的歧义。

### 1.2 集群架构

```
集群（Cluster）
  ├── Master节点（主节点）
  │     └── 负责：索引创建/删除、分片分配、集群状态
  ├── Data节点（数据节点）
  │     └── 负责：CRUD操作、数据存储
  └── Coordinating节点（协调节点）
        └── 负责：请求路由、结果聚合
```

**追问：节点角色可以同时兼具吗？**

可以兼任，但在生产环境建议分离：
- Master节点：不存储数据，避免脑裂影响数据
- Data节点：不作为Master，避免数据迁移时影响集群

## 二、倒排索引（深度追问）

### 2.1 正排 vs 倒排

| 类型 | 结构 | 查询方式 |
|------|------|----------|
| 正排索引 | 文档ID → 文档内容 | 逐行扫描，O(n) |
| 倒排索引 | Term → 文档ID列表 | 直接定位，O(1) |

**追问1：倒排索引的结构？**

```
倒排表（Posting List）:
["苹果"] → [doc1, doc5, doc9] → 文档ID列表
["香蕉"] → [doc2, doc7]       → 文档ID列表
["苹果手机"] → [doc3, doc8]    → 文档ID列表（短语）
```

**追问2：如何支持TF-IDF相关性排序？**

每个Term的倒排列表不仅存文档ID，还存储：
- TF（词频）：该词在文档中出现次数
- IDF（逆文档频率）：log(总文档数/包含该词的文档数)
- 文档长度归一化

### 2.2 倒排索引的写入过程

新增文档时：
1. 将文档加入内存缓冲区
2. 定时生成倒排索引（In-memory segment）
3. 定期刷到磁盘（Persistent segment）

**追问3：为什么ES写入这么慢？（对比Redis）**

ES是「写优读慢」的设计：
- 写入需要构建倒排索引（耗时）
- 读取只需查倒排表（快速）
- 权衡：牺牲写入性能，换取强大的查询能力

## 三、分片与副本（深度追问）

### 3.1 分片路由机制

```
shard_num = hash(_routing) % primary_shards
```

- 默认_routing = 文档的_id
- 可以自定义路由字段：`routing_field`

**追问1：为什么主分片数创建后不能改？**

因为路由算法依赖主分片数：
- 假设主分片=3，hash(id) % 3 = 1
- 改成5后，hash(id) % 5 可能等于 3，数据就找不到了

**追问2：如何重新设置主分片数？**

只能通过reindex重建索引：
```bash
POST _reindex
{
  "source": {"index": "old_index"},
  "dest": {"index": "new_index"}
}
```

### 3.2 副本机制

```
Primary Shard + N Replica Shard = N+1 份数据
```

**追问3：主分片和副本放在同一节点有问题吗？**

有问题：
- 节点宕机，所有副本都丢失
- 应该把副本分散到不同节点

**追问4：ES如何保证读写一致性？**

- **写操作**：必须写入主分片和所有副本同步完成后才返回
- **读操作**：可以读主分片或任一副本

## 四、写入流程（深度追问）

### 4.1 写入流程图

```
客户端 → 协调节点 → 主分片 → (并行) 副本 → 响应客户端
```

详细步骤：
1. 写入请求发送到协调节点
2. 协调节点计算目标主分片
3. 将请求转发给主分片节点
4. 主分片写入内存Buffer + Translog
5. 主分片同步到所有副本
6. 副本也写入内存 + Translog
7. 返回成功给协调节点

### 4.2 深入：Translog的作用

**追问1：为什么需要Translog？**

内存Buffer的数据可能丢失，需要记录操作日志：
- 数据先写入内存Buffer
- 同时写入Translog（落盘）
- 定期将Buffer刷成Segment

**追问2：Translog何时刷盘？**

- 每个写操作后同步刷盘（默认）
- 或者异步批量刷盘

### 4.3 Refresh vs Flush

| 操作 | 作用 | 触发条件 |
|------|------|----------|
| Refresh | Buffer → 新Segment（可搜索） | 默认1秒 |
| Flush | Translog → 磁盘（持久化） | 磁盘满或30分钟 |

**追问3：为什么ES搜索近实时？**

因为每秒Refresh一次，新数据在1秒后即可搜索。

**追问4：为什么ES写入不是实时持久化？**

数据先写内存和Translog，但没刷磁盘。 Flush才是真正的持久化。

## 五、搜索流程（深度追问）

### 5.1 Query-Then-Fetch流程

```
Query Phase:
协调节点 → 所有分片 → 执行查询，返回Top N → 协调节点合并

Fetch Phase:
协调节点 → 所有分片 → 获取完整文档 → 返回客户端
```

### 5.2 深度分页问题

**追问1：from+size为什么深分页会慢？**

```sql
-- 假设 from=10000, size=10
-- 每个分片要取 10010 条数据，协调节点要处理 N * 10010 条
```

问题：
- 每个分片独立返回from+size条数据
- 协调节点要排序所有分片的结果，取top

**追问2：scroll和search_after的区别？**

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| scroll | 游标，生成快照 | 一次性导出大量数据 |
| search_after | 上一页最后一条的排序值 | 实时分页，深度无限 |

**追问3：Point In Time (PIT) 是什么？**

ES 7.10+ 引入的改进版scroll：
- 自动维护索引快照
- 避免scroll的「数据过时」问题

```java
// 使用PIT分页
POST /my-index/_pit?keep_alive=5m  // 获取PIT ID

POST /_search
{
  "pit": {"id": "PIT_ID"},
  "size": 10,
  "search_after": [...]  // 游标
}
```

## 六、聚合分析（深度追问）

### 6.1 聚合类型

| 类型 | 说明 | 示例 |
|------|------|------|
| Bucket | 分桶 | 按城市分组 |
| Metric | 指标 | count, avg, sum |
| Pipeline | 聚合的聚合 | 环比增长 |
| Matrix | 矩阵（少用） | |

**追问1：聚合的性能问题？**

聚合默认只返回Top 10 bucket：
```json
{
  "aggs": {
    "city_agg": {
      "terms": {"field": "city", "size": 1000}  // 显式设置size
    }
  }
}
```

**追问2：为什么聚合默认只返回10个？**

节省内存和带宽，ES假设你只需要Top N。

## 七、性能优化（深度追问）

### 7.1 写入优化

**追问1：批量写入的大小？**

- 单次bulk建议5-15MB
- 文档数建议5000-10000条
- 需要通过压测调优

**追问2：refresh_interval设为-1会怎样？**

- 禁用自动refresh
- 写入速度最快
- 但新数据不可搜索
- 适合批量导入场景，导入完成后手动refresh

### 7.2 查询优化

**追问3：哪些查询尽量避免？**

| 查询类型 | 问题 | 替代方案 |
|----------|------|----------|
| wildcard | 性能差 | prefix、ngram |
| range + wildcard | 极差 | 复合查询 |
| deep paging | 内存高 | scroll/pit |

**追问4：routing的作用？**

```java
// 查询时指定routing，直接定位目标分片
GET my-index/_search?routing=user_123
```

- 减少搜索的分片数
- 提升查询性能
- 但增加了写入复杂度

### 7.3 索引设计优化

**追问5：冷热分离的原理？**

```
热数据（Hot）
├── 新数据/频繁查询
├── SSD存储
└── 高配置机器

冷数据（Cold）
├── 历史数据/少查询
├── HDD存储
└── 低配置机器
```

实现：
```bash
# 将旧索引移入cold节点
PUT my-index/_settings
{"index.routing.allocation.include.box_type": "cold"}
```
