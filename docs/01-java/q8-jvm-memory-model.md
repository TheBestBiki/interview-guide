# Q8: JVM内存模型深度剖析

> **特别说明**：JVM内存模型容易混淆两个概念：
> - **JVM运行时数据区**：JVM运行时的内存布局（哪些区域、存什么数据）
> - **JMM (Java Memory Model)**：Java多线程访问内存的抽象模型（volatile、synchronized的底层原理）
> 
> 本篇主要讲**运行时数据区**，并在最后对比JMM。

## 核心区域详解

### 1. 堆 (Heap) - 核心重灾区

```
存储内容：
- 对象实例（几乎所有）
- 数组
- Class对象
- String常量池（JDK 7及之前在PermGen，之后移入堆）

分区（JDK 8+）：
┌────────────────────────────────────────┐
│              堆内存                      │
├─────────────────────┬──────────────────┤
│      新生代         │     老年代        │
│  ┌─────┬─────┬───┐  │                   │
│  │Eden │ S0  │S1 │  │                   │
│  │ 80% │ 10% │10%│  │     60%           │
│  └─────┴─────┴───┘  │                   │
│      40%           │                   │
└─────────────────────┴──────────────────┘

特点：
- 线程共享，需要加锁
- OOM主要发生地
- 垃圾回收的主要工作区
- 可通过 -Xms -Xmx 配置
```

### 2. 元空间 (Metaspace) - JDK 8+ 重大变化

```
JDK 7及之前：PermGen（永久代）
┌─────────────────────────────────────────┐
│ PermGen (永久代)                        │
│ - 类信息                                │
│ - 常量池                               │
│ - 静态变量                             │
│ - JIT编译代码                          │
└─────────────────────────────────────────┘
问题：固定大小，容易OOM

JDK 8+：Metaspace（元空间）
┌─────────────────────────────────────────┐
│ Metaspace (元空间)                      │
│ - 类元数据                              │
│ - 方法元数据                           │
│ - JIT编译结果                          │
└─────────────────────────────────────────┘
特点：使用本地内存，动态扩展

配置参数：
-XX:MetaspaceSize=256m    # 初始大小
-XX:MaxMetaspaceSize=     # 无限制（默认）
```

### 3. 虚拟机栈 (VM Stack)

```
结构：每个线程一个栈
┌────────────────────────────────────────┐
│  栈帧 (Stack Frame)                    │
│  ┌──────────────────────────────────┐  │
│  │  局部变量表 (Local Variables)    │  │
│  │  - 参数                          │  │
│  │  - 局部变量                      │  │
│  │  - returnAddress (返回地址)      │  │
│  └──────────────────────────────────┘  │
│  ┌──────────────────────────────────┐  │
│  │  操作数栈 (Operand Stack)        │  │
│  │  - 计算中间结果                   │  │
│  │  - 指令操作数                    │  │
│  └──────────────────────────────────┘  │
│  ┌──────────────────────────────────┐  │
│  │  动态链接 (Dynamic Linking)      │  │
│  │  - 指向运行时常量池               │  │
│  └──────────────────────────────────┘  │
│  ┌──────────────────────────────────┐  │
│  │  方法返回地址 (Return Address)   │  │
│  └──────────────────────────────────┘  │
└────────────────────────────────────────┘

单位：栈帧
大小：-Xss1m (默认1MB)

异常：
- StackOverflowError：栈深度超限（递归没退出）
- OutOfMemoryError：栈内存不足（线程过多）
```

### 4. 本地方法栈 (Native Stack)

```
与虚拟机栈的区别：
- 虚拟机栈：执行Java方法
- 本地方法栈：执行Native方法（JNI）

HotSpot实现：
- 虚拟机栈和本地方法栈合一
- 使用相同的 -Xss 参数

其他JVM：
- 分离实现（如IBM J9）
- 独立配置（如 -Xoss）
```

### 5. 程序计数器 (Program Counter Register)

```
作用：
- 记录当前线程执行的字节码行号
- 执行Native方法时：undefined
- 执行Java方法时：字节码行号

特点：
- 线程私有
- 唯一不会发生OOM的区域
- CPU时间片切换后恢复执行位置
```

## 面试高频追问

### 追问1：为什么需要堆和方法区分离？

```
设计意图：
┌────────────────────────────────────────────────────┐
│  堆 (共享)           │  方法区 (共享)             │
│  - 对象实例          │  - 类信息（不变）           │
│  - 频繁创建/销毁     │  - 加载后不变               │
│  - 需要GC            │  - 一般不需要GC             │
└────────────────────────────────────────────────────┘

分离好处：
1. 不同的生命周期 → 不同的管理策略
2. 对象实例需要GC，类信息基本不需要
3. 内存效率最大化
4. 垃圾回收器可以专注堆
```

### 追问2：JDK 8为什么要移除PermGen？

```
PermGen的问题：
┌────────────────────────────────────────────┐
│ PermGen (固定大小)                         │
│ - -XX:MaxPermSize=64m                     │
│ - 难以调优                                 │
│ - 容易OOM: PermGen space                  │
│ - 类加载过多就满                           │
│ - 浪费内存：即使释放也不会还给我们         │
└────────────────────────────────────────────┘

Metaspace的优势：
┌────────────────────────────────────────────┐
│ Metaspace (动态扩展)                       │
│ - 使用本地内存（Native Memory）            │
│ - 默认无上限                               │
│ - 释放后归还操作系统                       │
│ - 类卸载能力                               │
│ - 更灵活                                   │
└────────────────────────────────────────────┘

注意：Metaspace OOM 会抛出：
java.lang.OutOfMemoryError: Metaspace
```

### 追问3：对象在堆中的分配过程？

```
分配流程：
┌─────────────────────────────────────────────────┐
│  1. TLAB分配 (Thread Local Allocation Buffer) │
│     每个线程预分配Eden区一小块                   │
│     └── 线程私有，无需加锁                      │
│              ↓                                  │
│  2. Eden区分配                                  │
│     ├── 有空间 → 分配成功                       │
│     └── 空间不足 → 触发Minor GC                 │
│              ↓                                  │
│  3. 老年代分配 (大对象/分配担保)                 │
│     ├── -XX:PretenureSizeThreshold=1MB         │
│     └── 直接在Old区分配                         │
│              ↓                                  │
│  4. 分配失败 → Full GC                         │
│              ↓                                  │
│  5. 还失败 → OOM                               │
└─────────────────────────────────────────────────┘
```

### 追问4：String常量池在哪里？

```
JDK 6 及之前：
┌─────────────────────────────────────┐
│  PermGen (永久代)                   │
│  ┌─────────────────────────────┐    │
│  │  String Constant Pool      │    │
│  │  (字符串常量池)              │    │
│  └─────────────────────────────┘    │
└─────────────────────────────────────┘

JDK 7：
┌─────────────────────────────────────┐
│  堆 (Heap)                          │
│  ┌─────────────────────────────┐    │
│  │  String Constant Pool      │    │
│  │  (字符串常量池)              │    │
│  └─────────────────────────────┘    │
│  - 从PermGen移到堆                  │
│  - 受GC管理                          │
└─────────────────────────────────────┘

intern() 方法变化：
- JDK 6: 始终在PermGen创建
- JDK 7+: 先检查堆常量池，没有才创建
```

### 追问5：直接内存 (Direct Memory) 是什么？

```
定义：JVM进程在Native分配的内存，不受JVM堆管理

使用场景：
┌────────────────────────────────────────┐
│  NIO (New I/O)                        │
│  ┌──────────────────────────────────┐ │
│  │  ByteBuffer.allocateDirect()    │ │
│  │  - 堆外内存                      │ │
│  │  - 零拷贝                       │ │
│  │  - 高性能网络/文件IO            │ │
│  └──────────────────────────────────┘ │
└────────────────────────────────────────┘

配置：
-XX:MaxDirectMemorySize=1g   # 直接内存最大

优势：
- 减少堆内存占用
- 减少GC压力
- IO操作零拷贝

注意：
- 不受GC管理，需要手动释放
- 容易导致OOM: Direct buffer memory
```

### 追问6：栈帧包含哪些内容？

```
栈帧结构：
┌────────────────────────────────────────┐
│  局部变量表 (Local Variables)          │
│  ┌──────────────────────────────────┐ │
│  │ slot 0: this (非静态方法)       │ │
│  │ slot 1: 第一个参数              │ │
│  │ slot 2: 第二个参数              │ │
│  │ ...                             │ │
│  │ slot n: 局部变量                │ │
│  │ - 基本类型：1个slot            │ │
│  │ - long/double：2个slot         │ │
│  │ - 对象引用：1个slot             │ │
│  └──────────────────────────────────┘ │
│  大小：-XX:LocalVariableTableCheck     │
├────────────────────────────────────────┤
│  操作数栈 (Operand Stack)              │
│  - 表达式求值                           │
│  - 最大深度：编译时确定                  │
│  - 典型操作：dup, swap, iadd, ifeq    │
├────────────────────────────────────────┤
│  动态链接 (Dynamic Linking)            │
│  - 符号引用 → 直接引用                  │
│  - 方法调用指令 → 实际方法               │
├────────────────────────────────────────┤
│  方法返回地址 (Return Address)          │
│  - 方法正常返回                          │
│  - 方法异常返回                          │
└────────────────────────────────────────┘
```

### 追问7：什么是TLAB？

```
TLAB (Thread Local Allocation Buffer)

背景：
- 堆是线程共享
- 对象分配需要同步（CAS）
- 竞争激烈影响性能

解决方案：
┌─────────────────────────────────────────┐
│  每个线程预分配一小块Eden区              │
│  ┌────────────────────────────────────┐ │
│  │  Thread 1 TLAB    │  Thread 2 TLAB │ │
│  │  [=====]          │  [=====]        │ │
│  │  Eden区          │  Eden区          │ │
│  └────────────────────────────────────┘ │
└─────────────────────────────────────────┘

配置参数：
-XX:+UseTLAB           # 启用TLAB（默认）
-XX:TLABSize=50k       # TLAB大小
-XX:TLABRefillWasteThreshold=1k  # 浪费阈值

优点：
- 分配无需加锁
- 减少竞争
- 提高性能
```

### 追问8：运行时常量池 vs 字符串常量池

```
容易混淆的两者：

┌─────────────────────────────────────────────────┐
│  运行时常量池 (Runtime Constant Pool)          │
│  - 属于方法区                                   │
│  - 类加载后，class文件常量池进入方法区           │
│  - 包含：                                       │
│    • 字面量 (字面量)                            │
│    • 符号引用 (方法/字段引用)                   │
│    • 方法句柄                                   │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│  字符串常量池 (String Constant Pool)            │
│  - 属于堆（JDK 7+）                             │
│  - 存储字符串字面量                             │
│  - intern()方法使用                             │
│  - 类似一个Map<String, String>                 │
└─────────────────────────────────────────────────┘

关系：
- 字符串常量池是运行时常量池的一部分
- 运行时常量池包含更多类型的常量
```

### 追问9：JVM内存模型 vs JMM (Java Memory Model)

```
这是两个完全不同的概念！

┌─────────────────────────────────────────────────────────┐
│  JVM内存模型 (JVM Memory Model)                         │
│  = JVM Runtime Data Areas                              │
│  回答：JVM运行时，数据在哪里存储？                      │
│  - 堆、栈、方法区、本地方法栈、PC寄存器                 │
│  - 物理层面的内存布局                                   │
│  - 面试常问：内存溢出、内存分配                         │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  JMM (Java Memory Model)                               │
│  回答：多线程如何访问共享变量？                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │  线程A                    线程B                  │   │
│  │  ┌───────┐              ┌───────┐              │   │
│  │  │工作内存│ ← ── → │工作内存│              │   │
│  │  └───────┘    ↑     └───────┘              │   │
│  │       │       │         │                   │   │
│  │       └───────┼─────────┘                   │   │
│  │               ↓                             │   │
│  │       ┌───────────────────┐                 │   │
│  │       │    主内存 (Main)   │                 │   │
│  │       │  - 堆中的实例      │                 │   │
│  │       │  - 方法区中的数据  │                 │   │
│  │       └───────────────────┘                 │   │
│  └─────────────────────────────────────────────────┘   │
│  - volatile、synchronized底层原理                     │
│  - happens-before 规则                                 │
│  - 解决可见性、有序性问题                               │
└─────────────────────────────────────────────────────────┘
```

### 追问10：内存溢出(OOM)有哪些类型？

```
常见OOM类型：

1. java.lang.OutOfMemoryError: Java heap space
   - 堆内存不足
   - 原因：对象太多、内存泄漏

2. java.lang.OutOfMemoryError: Metaspace
   - 元空间不足
   - 原因：类加载过多

3. java.lang.OutOfMemoryError: Direct buffer memory
   - 直接内存不足
   - 原因：NIO使用过多

4. java.lang.OutOfMemoryError: unable to create native thread
   - 线程数量过多
   - 原因：创建大量线程

5. java.lang.OutOfMemoryError: GC overhead limit exceeded
   - GC开销超限
   - 原因：频繁GC但回收效果差

6. java.lang.OutOfMemoryError: PermGen space (JDK 7-)
   - 永久代不足
   - 原因：类加载过多（JDK 7-）

栈溢出：
- java.lang.StackOverflowError
  原因：递归调用太深

- java.lang.OutOfMemoryError: native stack overflow
  原因：Native方法调用过深
```

## JVM参数速查表

```bash
# 堆内存
-Xms4g                  # 初始堆大小
-Xmx4g                  # 最大堆大小
-Xmn2g                  # 新生代大小
-XX:NewRatio=2          # 老年代/新生代=2

# 元空间
-XX:MetaspaceSize=256m  # 初始元空间
-XX:MaxMetaspaceSize=   # 最大（默认无限制）

# 栈
-Xss1m                  # 线程栈大小
-Xss512k                # 更小的栈

# 直接内存
-XX:MaxDirectMemorySize=1g

# TLAB
-XX:+UseTLAB            # 启用TLAB
-XX:TLABSize=50k        # TLAB大小

# 其他
-XX:SurvivorRatio=8     # Eden/Survivor=8
-XX:PretenureSizeThreshold=1m  # 大对象阈值
```

## 面试加分总结

> "JVM内存模型是Java开发者的基本功。我把它分成两部分理解：
> 
> **运行时数据区**是物理层面的：堆存对象、栈存方法调用、方法区存类信息。这里面试最常问的是堆内存划分和JDK 8移除PermGen的原因。
> 
> **JMM**是抽象层面的：解决多线程可见性和有序性问题。面试官常通过volatile、synchronized来考察你对这个的理解。
> 
> 很多人容易混淆这两个概念，我面试时会主动说明，避免被带偏。"
