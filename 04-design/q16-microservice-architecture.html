Q16: 如何设计一个可扩展的微服务架构？ ​

架构设计原则 ​

原则说明
单一职责每个服务专注单一业务域
高内聚低耦合服务内部紧耦合，服务间松耦合
服务自治独立部署、测试、升级
可观测日志、监控、链路追踪
基础设施自动化CI/CD、容器化、自动扩缩容

领域驱动设计（DDD） ​

DDD分层架构 ​

API Gateway              ← 网关层
Application Service     ← 应用层（编排、事务）
Domain Service / Entity ← 领域层（核心业务逻辑）
Repository / DAO        ← 基础设施层（持久化）

DDD战略建模 ​

  • 子域（SubDomain）：业务领域划分
  • 限界上下文（Bounded Context）：服务边界
  • 实体（Entity）：有唯一标识的对象
  • 值对象（Value Object）：无标识的属性集合

服务拆分策略 ​

按业务能力拆分 ​

  • 订单服务：订单创建、查询、取消
  • 支付服务：支付、退款
  • 库存服务：库存查询、扣减
  • 用户服务：注册、登录、权限

拆分粒度原则 ​

  • 团队规模：2 Pizza Team（6-8人）负责一个服务
  • 独立演进：每个服务可独立部署和升级
  • 数据私有：服务间通过API交互，不共享数据库

监控与链路追踪 ​

监控体系：
├── 基础设施监控：Prometheus + Grafana
├── 应用监控：SkyWalking / Jaeger
├── 日志收集：ELK / Loki
└── 告警：AlertManager + 钉钉/邮件

架构演进路线 ​

  1. 单体架构：所有功能在一个应用中，适合小团队
  2. 垂直拆分：按业务拆分为多个应用
  3. 服务化：引入服务注册发现、配置中心、熔断降级
  4. 微服务：领域驱动设计、容器化部署（K8s）
